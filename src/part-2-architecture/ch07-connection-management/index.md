# Chapter 7: Connection Management

In traditional networking, establishing a connection is almost trivially simple. A TCP client calls `connect()`, the kernel performs a three-way handshake, and the application receives a connected socket ready for data transfer. The entire process is hidden behind a single blocking call. The programmer never thinks about the state machine inside the kernel, the sequence number negotiation, or the path MTU discovery that happens transparently beneath the API.

RDMA connection management is a different world entirely. Establishing a connection between two RDMA endpoints requires the application to explicitly navigate a multi-state machine, exchange endpoint information through an out-of-band mechanism, and configure dozens of transport parameters that the hardware needs before it can send or receive a single byte. This complexity is not accidental -- it is a direct consequence of the design principles that make RDMA fast. Because the NIC handles transport processing autonomously, without kernel involvement on the data path, all the parameters that a traditional kernel would negotiate dynamically must be configured upfront and programmed into the hardware before data transfer begins.

Consider what happens when two nodes want to exchange data over RDMA using Reliable Connected (RC) transport. Each side must create a Queue Pair, transition it through a precise sequence of states -- RESET to INIT to RTR (Ready to Receive) to RTS (Ready to Send) -- and at each transition, supply the correct set of attributes. The INIT-to-RTR transition requires knowing the remote side's QP number, the path MTU, the destination LID or GID -- information that can only come from the remote node. This means the two sides must exchange metadata before they can even begin the connection process. Unlike TCP, where the kernel handles this exchange transparently, RDMA applications must arrange it themselves, whether through a TCP sideband channel, a shared configuration service, or the InfiniBand Communication Manager.

This chapter covers the full landscape of RDMA connection management, from low-level mechanisms to high-level abstractions.

**Section 7.1: QP State Machine** dissects the Queue Pair state machine in detail. Every QP type -- RC, UC, UD -- follows a state machine that governs what operations are legal at each point. We examine every state, every transition, the exact attributes required at each step, and the common mistakes that cause `ibv_modify_qp()` to fail with cryptic error codes.

**Section 7.2: Communication Manager (CM)** explores the InfiniBand Communication Manager, the kernel-level protocol that automates connection establishment using a three-way handshake (REQ/REP/RTU) carried over management datagrams. The CM handles the metadata exchange that applications would otherwise need to arrange manually, and provides mechanisms for service discovery, alternate path migration, and graceful disconnection.

**Section 7.3: RDMA_CM** introduces the RDMA Connection Manager library (`rdma_cm`), a user-space abstraction that dramatically simplifies connection management by providing a socket-like event-driven API. The RDMA_CM handles address resolution, route resolution, QP state transitions, and connection lifecycle management, reducing hundreds of lines of boilerplate to a handful of function calls.

**Section 7.4: Connection Patterns** addresses the architectural challenge of connection management at scale. When a cluster grows beyond a handful of nodes, the naive approach of creating a dedicated RC Queue Pair for every peer produces an N-squared explosion in resource consumption. We examine the problem quantitatively and survey the transport-level solutions -- Unreliable Datagram, eXtended Reliable Connected (XRC), Dynamically Connected Transport (DCT), and Shared Receive Queues -- that the InfiniBand architecture provides to tame this combinatorial growth.

Connection management is where many RDMA newcomers stumble. The APIs are unforgiving: a single missing attribute flag in a QP transition call produces a silent failure or a kernel log message that offers little diagnostic value. Understanding the mechanics deeply -- not just the happy path, but the error cases and recovery procedures -- is essential for building RDMA applications that are robust in production. By the end of this chapter, you will have a thorough understanding of how RDMA connections are established, managed, and scaled.
