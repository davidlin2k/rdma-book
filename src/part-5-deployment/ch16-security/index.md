# Chapter 16: Security

RDMA's security model is fundamentally different from traditional networking, and understanding these differences is essential for anyone deploying RDMA in production environments. In conventional TCP/IP networking, security has been layered on over decades: firewalls inspect and filter packets, TLS encrypts data in transit, and operating system kernels mediate all access to network resources. Applications never touch the network hardware directly, and every byte of data passes through multiple security checkpoints before reaching its destination.

RDMA dismantles this entire model in the name of performance. Kernel bypass means the application talks directly to the network interface card, circumventing the operating system's security enforcement. Zero-copy means data moves from one machine's memory to another without intermediate buffering where inspection could occur. One-sided operations mean a remote machine's memory can be read or written without any software on that machine being notified, let alone having the opportunity to approve or deny the access.

This is not to say that RDMA is inherently insecure. Rather, its security model operates on different principles. Instead of inspecting every packet and mediating every operation through software, RDMA relies on hardware-enforced access controls: protection domains isolate resources between applications, memory keys authorize specific memory regions for remote access, and partition keys segment the network at the fabric level. These mechanisms are efficient precisely because they are checked in hardware, but they are also less flexible and less familiar than the software-based security that most engineers are accustomed to.

The challenge is compounded in modern cloud and multi-tenant environments. When multiple untrusted applications share the same physical network and potentially the same physical NIC, the security boundaries must be airtight. A single vulnerability in memory key management could expose one tenant's data to another. A misconfigured protection domain could allow unauthorized memory access across application boundaries.

This chapter examines RDMA security from the ground up. We begin with the trust model and how it differs from traditional networking. We then explore the specific access control mechanisms -- memory keys, protection domains, and partition keys -- and how to use them correctly. We survey the known attack surfaces, from remote memory scanning to denial-of-service vectors, and the mitigations that have been developed. Finally, we provide concrete guidance for secure deployment, covering network isolation, application-level authentication, and the emerging technologies that promise to close the remaining security gaps.

Security in RDMA is an evolving area. As RDMA moves beyond trusted HPC clusters into broader data center deployments, the community is actively working on stronger protections. Understanding the current state of the art -- both its strengths and its limitations -- is the first step toward deploying RDMA safely.
